\section{Modelagem dos Agentes}
\label{sec:modelagem}

\todo[inline]{Completar a modelagem dos agentes}

Como o eProfile é um modelo onde é necessário inferir informações de forma autônoma a respeito de dados armazenados em uma ontologia, a tecnologia de agentes autônomos foi escolhida para o desenvolvimento. A metodologia utilizada para a modelagem é a Prometheus \cite{prometheus}. Esta metodologia define um processo detalhado para a especificação, projeto, implementação e teste/depuração de sistemas de software orientados a agentes. \cite{prometheus-book}

\subsection{Especificação do Sistema}
\label{sec:especificacao}

\begin{figure}
  \includegraphics[width=0.8\textwidth]{imagens/prometheus.png}
  \caption[Metodologia Prometheus]{Processo de projeto segundo a metodologia Prometheus \cite{prometheus-book}}
  \label{fig:metodologia}
\end{figure}

A metodologia Prometheus, ilustrada na Figura~\ref{fig:metodologia} define que o primeiro passo da modelagem de um sistema é a identificação dos objetivos inciais do sistema. De acordo com o modelo, o eProfile é um sistema que gerencia perfis de entidades, permitindo que as aplicações definam (i) seus próprios modelos de usuário e (ii) regras através das quais a trilha de uma entidade será processada. Este processo de inferência monta ou atualiza um perfil de entidade, que é então disponibilizado para a aplicação.

A partir desta descrição, é possível identificar os objetivos e seus respectivos sub-objetivos, que são definidos considerando a forma como cada objetivo pode ser atingido \cite{prometheus-book}. Assim, os objetivos e sub-objetivos são:

\begin{itemize}
  \item Gerenciar múltiplos perfis de entidade.
  \begin{itemize}
    \item Permitir que aplicações se identifiquem;
    \item Aceitar requisições de novas entidades;
    \item Disponibilizar os perfis inferidos à aplicação.
  \end{itemize}
  \item Permitir que as aplicações definam seus próprios modelos de usuário.
  \begin{itemize}
    \item Receber novos modelos de usuário da aplicação;
    \item Validar os modelos de usuário;
    \item Armazenar os modelos de usuário.
  \end{itemize}
  \item Permitir que as aplicações definam suas regras de inferência.
  \begin{itemize}
    \item Receber as regras;
    \item Validar as regras;
    \item Armazenar as regras.
  \end{itemize}
  \item Buscar as trilhas de um usuário a partir de um gerenciador de trilhas.
  \begin{itemize}
    \item Interagir com os gerenciadores de trilhas suportados;
    \item Converter as informações do formato de trilha original para a ontologia de trilhas do eProfile.
  \end{itemize}
  \item Inferir a trilha, montando um perfil de entidade.
  \begin{itemize}
    \item Agendar a execução das regras;
    \item Executar a inferência de acordo com as regras;
    \item Gerenciar sobrecarga do servidor e reagendar as regras, se necessário.
  \end{itemize}
\end{itemize}

É necessário também a identificação das bases de dados necessárias para o funcionamento do sistema. O eProfile faz uso das seguintes bases de dados:

\begin{itemize}
  \item Base de dados de modelos de usuário -- contém os modelos de usuário registrados pela aplicação;
  \item Base de dados de regras de inferência -- contém as regras de inferência que foram enviados pela aplicação;
  \item Base de dados de trilhas -- contém as trilhas buscadas do gerenciador de trilhas, e convertidas para o formato do eProfile;
  \item Base de dados de perfis de entidades -- contém os perfis de entidades, já inferidos a partir dos modelos, regras e trilhas.
\end{itemize}

\subsection{Conversor}
\label{sec:conversor}

O \textbf{agente conversor} é responsável por realizar a conversão dos dados da trilha do gerenciador de trilhas (externo) para a ontologia de trilha do eProfile (interno). A trilha do gerenciador pode ser disponibilizada em qualquer formato e por qualquer meio, desde que o eProfile ofereça suporte ao gerenciador de trilhas específico. O agente conversor tem também a responsabilidade de verificar, a cada período pré-determinado, atualizações nos dados disponibilizados no gerenciador.

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%

\subsection{Regulador}
\label{sec:regulador}

O \textbf{agente regulador} realiza o recebimento das regras enviadas pela aplicação. Uma vez que as regras sejam recebidas, elas são validadas e, se forem consideradas válidas, são armazenadas na base de regras. A aplicação é então informada do aceite ou rejeição. O agente pode rejeitar uma regra se: (i) a regra for inválida ou (ii) se for verificado que a execução da regra irá consumir recursos excessivos do servidor.

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%

\subsection{Raciocinador}
\label{sec:raciocinador}

O \textbf{agente raciocinador} é responsável pela inferência das trilhas de acordo com as regras enviadas pela aplicação, e posterior atualização do perfil da entidade. A execução do agente não necessita que o cliente ou o gerenciador de trilhas esteja conectado, e ocorre da seguinte forma:

\begin{enumerate}

\item O agente verifica as regras, comparando a data da útlima execução da regra com a freqüência de execução, definida pela aplicação na criação da regra.

\item Caso seja o momento da execução da regra, a mesma é executada através da biblioteca externa HermiT \cite{motik09}.

\item Caso a regra tenha sido executada com sucesso, o resultado da regra é atualizado no perfil. Em caso de insucesso, o erro é enviado para a aplicação no momento que a mesma se reconectar.

\end{enumerate}
